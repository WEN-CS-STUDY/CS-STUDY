### HTTPS

- 기존 HTTP에 Security (보안) 기능을 붙인 것
    - 접속한 사이트가 믿을만한 사이트인지 확인
    - 서버로 데이터를 전송할 때 데이터를 암호화해준다.

### HTTP

- 서로 다른 시스템들 사이에서 통신을 주고 받게 해주는 가장 기초적인 프로토콜
- 문제점 중 하나가 서버에서부터 브라우저로 전송되는 정보가 암호화되지 않는다는 점이다.

### 원리

- 공개키 암호화 방식 (데이터 위변조)
- 전자 서명 방식 (신뢰할 수 있는 사이트)

### 실제 구현

- 네이버의 공개키를 사용자들에게 뿌려서 데이터를 암호화 하는데 이거 진짜 네이버 키가 맞음?
    - 이걸 인증해주는 CA라는 기관이 있다.
- 과정
    - 브라우저 (클라이언트)는 아직 서버를 신뢰하지 못한다.
    - handshake과정을 수행한다. 이 때 성공하면 서버의 인증서를 클라이언트에게 전송한다.
    - 클라이언트는 CA를 통해 인증서가 진짜인지 확인한다.
        - 인증서는 CA의 개인키로 암호화되어 있고, CA의 공개키로 이것을 풀 수 있다.
    - 인증서가 위조되지 않고 성공적으로 복호화되면 이 인증서에는 서버의 공개키가 포함되어 있다.
    - 이후에 데이터를 주고받을 때는 비대칭키로 대칭키를 암호화해서 전달하여 이 대칭키로 통신한다.
    - 이후 세션이 종료되면 해당 비대칭키는 폐기된다.

### 장점

- 데이터 위변조 방지, 신뢰할 수 있는 사이트로 보안 강화
